# Storage Contract Matrix

## Purpose / Scope

This document is a descriptive-first, docs-only matrix of currently observed storage surfaces in the Brunch Body repo. Current code is the source of truth. The matrix records observed storage surfaces and bounded owner labels; it does not change runtime behavior.

Migration, reset/delete semantics, and import/export behavior changes are deferred to later lanes. Where a row's strongest architectural claim is not directly declared in code, that row stays in the matrix but uses an `inferred` or `unknown` evidence label instead of overstating certainty.

## Legend

- Owner labels:
  - `canonical owner`: current code directly supports this storage surface as the bounded owner for the row being documented.
  - `compatibility read`: current code still reads this direct storage key, but ownership appears to overlap with a persisted Redux slice and later normalization is deferred.
  - `sidecar/reference store`: current code uses this surface as adjacent reference data rather than the main persisted Redux root.
  - `unknown / needs follow-on`: current code shows overlapping or timing-sensitive ownership, and this lane does not settle the contract.
- Evidence labels:
  - `repo-observed`: the row's strongest architectural claim is directly supported by current code.
  - `inferred`: the row is required and code-observed, but the owner classification depends on combining multiple files such as direct reads plus persisted-slice overlap or bootstrap timing.
  - `unknown`: the row is included, but current code does not prove the architectural claim.
- Evidence labels apply to the row's strongest architectural claim, not just to the existence of the key.
- A row may use multiple supporting references when owner classification depends on direct storage reads plus persisted-slice overlap or bootstrap timing.
- For onboarding draft rows, `canonical owner` means canonical owner of the draft key itself, not of the broader auth/profile domain.
- Migration status values used here:
  - `current-contract`: this lane treats the current bounded contract as active and explicitly documented.
  - `deferred`: current behavior is observed, but normalization or migration decisions are intentionally left to follow-on lanes.

## Trust / Risk Notes

- Several domains dispatch into persisted Redux slices while still bootstrapping from direct storage keys. This matrix records that overlap; it does not normalize it.
- This is not a full persistence register. Adjacent seams such as `local_password`, `local_password_reset_requested_at`, `exercise_directory`, and `is_initialized` are intentionally out of the primary matrix for this lane.
- Migration, reset/delete semantics, and import/export behavior changes remain deferred. Row notes call them out only when directly relevant to the observed contract surface.
