# Persistence Inventory

This document is a present-state, docs-only inventory of how Brunch Body currently persists local data as inspected on April 17, 2026. It is intended to give later storage-contract, migration, and deletion-semantics lanes a repo-truthful baseline without changing current behavior.

- What changed: one internal architecture inventory document.
- What users will experience: no app behavior change.
- What docs or disclosures must be updated now: none in this lane, though mismatches are flagged below for later review.

## Method And Evidence Labels

This inventory was built from a repo-wide search for persistence APIs, wrappers, export or import file flows, and clear or reset touchpoints. The named storage and settings files were treated as a minimum audit seed, not an exhaustive boundary.

Evidence rules used here:

- `repo-observed`: directly proven by current repo code.
- `inferred`: reasonable conclusion that depends partly on framework or library convention, indirect evidence, or disclosure-only context.
- `unknown`: plausible but not proven from the inspected repo.

Additional audit rules:

- Live code wins when docs disagree.
- Package manifest presence alone is not counted as active usage.
- Commented-out code is not counted as an active persistence path.
- Absence claims mean "no repo-observed path was found in the repo-wide search"; they are not runtime guarantees.
- Where a storage entry name is library-conventional rather than directly authored in app code, the conventional portion is labeled `inferred`.

## Current Persistence Topology

The current local persistence story is split across four live surfaces:

1. One Redux root persisted store backed by AsyncStorage.
   - `src/redux/store/store.js` defines one `redux-persist` config with `key: 'root'`, AsyncStorage as the backend, and a whitelist of `auth`, `recreation`, `journal`, `nutrition`, `calendar`, `exercise`, and `todo`.
   - `src/root-container/RootContainer.js` mounts that store under `<PersistGate>`.

2. Direct AsyncStorage keys outside the persisted root.
   - Auth and onboarding flows read and write `user_profile`, local password keys, and onboarding draft keys directly.
   - Several feature domains still read direct AsyncStorage keys during bootstrap-style `GET_*` actions even though the same domain state also lives inside the persisted Redux root.

3. One MMKV sidecar store for bundled recreation plans.
   - `src/storage/mmkv/index.js` creates the `workout-storage` MMKV instance.
   - `src/storage/mmkv/hydration.js` seeds `plans_brunch_body` and `is_initialized`.
   - Recreation actions read bundled plans from MMKV through `getJSON(...)`.

4. One local file export flow.
   - `src/screens/setting/pages/Export To CSV/ExportToCSV.js` builds an `.xlsx` workbook from journal state and writes it to a user-selected document tree through `react-native-scoped-storage` and `react-native-fs`.

Important current boundary notes:

- The first route gate is not Redux rehydration. `AppBootstrap` checks direct `user_profile` presence before `RootContainer` mounts `<PersistGate>`.
- `user_profile` is duplicated across direct AsyncStorage and the persisted Redux `auth` slice.
- `logout`, `RESET_APP`, and `deleteAccount` are distinct current behaviors and should not be collapsed into one "clear data" semantic bucket.
- This audit found no repo-observed import or restore flow that complements the export screen.

## Trust And Disclosure Relevance

### What data appears to persist locally today

- Persisted Redux state for `auth`, `recreation`, `journal`, `nutrition`, `calendar`, `exercise`, and `todo`.
- Direct AsyncStorage keys for local profile, onboarding drafts, local password state, journal traits, and several domain bootstrap reads.
- MMKV entries for bundled Brunch Body plan hydration and initialization state.
- User-selected exported journal workbook files written outside the app's managed state.

### What clearing surfaces exist today

- `logout`: scoped removal of direct auth and onboarding-draft keys plus Redux auth-slice clearing.
- `RESET_APP`: in-memory Redux reset only.
- `deleteAccount`: full AsyncStorage clear plus full MMKV clear, immediately followed by bundled-plan MMKV rehydration.

### Observed trust or disclosure mismatches worth carrying forward

- Delete-account wording does not match the actual local-only implementation.
  - The app has no repo-observed backend account deletion path.
  - The current flow is a device-local reset with bundled-plan rehydration, not a remote account delete.
  - Evidence: `src/screens/setting/components/My Profile/DeleteAccount.js:43-116`, `src/screens/setting/pages/MyProfile/DeleteAccount.js:31-67`, `src/redux/actions/auth.js:132-156`.

- Export naming does not match the current file format.
  - Settings and release wording say "Export to CSV" or "Export to CSV" style language.
  - The implementation writes `.xlsx` files with Excel MIME type.
  - Evidence: `src/screens/setting/pages/Setting/Setting.js:107-115`, `src/screens/setting/pages/Export To CSV/ExportToCSV.js:104-139`, `docs/release/RELEASE_NOTES_1.0.3-rc.2.md:12-18`.

- README storage wording is directionally useful but not fully aligned with current code detail.
  - README groups `traits` with startup/bootstrap reads, while live initial-route gating uses direct `user_profile` presence in `AppBootstrap`.
  - README also lists direct domain keys without distinguishing that several are repo-observed read seams with no matching direct write path in current app code.
  - Evidence: `README.md:89-116`, `src/bootstrap/AppBootstrap.js:12-20`, `src/redux/actions/{journal,nutrition,calendar,exercise,recreation,todo}.js`.

- Privacy-manifest UserDefaults declaration is disclosure context, not a mapped Brunch Body-owned data row in this audit.
  - `ios/BrunchBody/PrivacyInfo.xcprivacy` declares required-reason access for `NSPrivacyAccessedAPICategoryUserDefaults`.
  - This audit did not find an app-owned native persistence surface that cleanly maps a Brunch Body user-data contract onto that declaration, so it should be treated as contextual or library-level evidence, not as a separate confirmed storage row; the exact Brunch Body-owned data mapping is `unknown` from current repo evidence.
  - Evidence: `ios/BrunchBody/PrivacyInfo.xcprivacy:5-15`.
